# Agent Outcome Benchmark — Protocol

**Phase P.1** of `docs/goal-completion-plan.md`.

## Purpose

Measure whether polyflow's graph actually improves an AI agent's ability to
identify impact scope — in tokens, wall time, and ground-truth recall — and
whether the semantic (vector) search arm adds value over FTS-only.

A null or negative result is equally valuable: it redirects Tier priorities
instead of shipping overconfidence.

## Three arms (pinned)

| # | Name | Description |
|---|------|-------------|
| 1 | `with_polyflow_semantic` | `claude -p` with polyflow MCP registered; index built with embeddings active |
| 2 | `with_polyflow_fts_only` | Same MCP config; index was built with `--no-embed` so `Search` runs FTS-only and `semantic: "unavailable: …"` |
| 3 | `without_polyflow` | `claude -p --strict-mcp-config` with no MCP servers; agent answers from scratch |

**Arm 2 toggle:** `polyflow index --no-embed` stamps `embed_status = "unavailable: embeddings skipped"`;
`Search` reads this via `GetEmbedStatus` and degrades to FTS-only automatically. No config change or
binary rebuild needed. The bench command temporarily sets this flag in the live DB and restores it.

## Tasks

- **Source:** impact cases (`kind: node` or `kind: file`) from every corpus manifest under `eval/corpus/`.
- **Prompt template (node):** `"In the <repo> codebase, if <target> were modified or renamed, which source files would need to be updated as a result? List each file path on its own line."`
- **Prompt template (file):** `"In the <repo> codebase, if the file <target> were modified, which other source files would be affected? List each file path on its own line."`
- **Answer key:** `expected_impacted` and `must_not_miss` from the corpus manifest (hand-verified ground truth; never generated by polyflow).

Up to 10 tasks per corpus repo are used; the corpus currently supplies the polyflow self-referential cases
(3 tasks) and any local corpora that are indexed.

## Metrics (per task/trial)

| Metric | How measured |
|--------|-------------|
| `input_tokens` | `usage.input_tokens` from claude JSON output |
| `output_tokens` | `usage.output_tokens` from claude JSON output |
| `wall_ms` | Local `time.Since(start)` wrapping the `claude -p` invocation |
| `total_cost_usd` | `total_cost_usd` from claude JSON output |
| `recall` | `\|agent-named files ∩ expected\| / \|expected\|` via `eval.Score` |
| `precision` | `\|agent-named files ∩ expected\| / \|agent-named files\|` |
| `hard_fail` | Any `must_not_miss` file absent from agent's answer |

**File extraction:** file paths in the agent's answer are extracted by `agentbench.ExtractFiles` (regex over
recognized extensions; deduplicated, sorted). The same `eval.Score` function used in `polyflow eval` is
applied so the recall numbers are directly comparable.

## Trials

3 trials per task/arm is the target (variance estimation). The first committed run uses 1 trial per task
to keep token cost low; subsequent runs should increase to 3. All deviations are recorded in the run's
`note` field.

## Output

- `eval/agent-bench/results/<date>.json` — machine-readable full results
- `eval/agent-bench/results/<date>.md` — human-readable summary table

Runs are **manual-triggered** (they cost real tokens) and **never in CI**.

## Running

```bash
# Full benchmark (all 3 arms, 1 trial each):
polyflow bench

# Single arm (faster, for iteration):
polyflow bench --arm with_polyflow_semantic

# Dry run (see tasks without calling claude):
polyflow bench --dry-run

# More trials:
polyflow bench --trials 3
```

## Interpreting results

- **Recall gap between arm 1 and arm 3** measures the graph's actual contribution.
- **Recall gap between arm 1 and arm 2** measures what semantic search adds over FTS.
- **Token savings** between arm 3 and arm 1 measure whether the graph reduces the agent's search cost.
- A null result on any axis is recorded honestly and redirects priorities.
